PROTO-01 WORKING PROTOTYPE

Inbox Triage & Auto-Draft Copilot

Reads an inbound ops request, classifies it by category and business-impact urgency (not tone), flags sentiment, and drafts a reply a reviewer can send with light editing instead of writing from scratch.

CORE: Claude API, single prompt DESIGNED FOR: n8n / Power Automate webhook → Airtable → Slack
// Problem

Every inbound request — IT, HR, finance, vendor, facilities — gets read, categorized, and prioritized by a person before anything happens, and urgency calls are inconsistent depending on who's triaging that day.

// Designed integration
FORM → N8N WEBHOOK → CLAUDE (CLASSIFY + DRAFT) → AIRTABLE LOG → SLACK ALERT

The Claude step is what's actually running below. The form intake and Airtable/Slack legs are the designed integration — not connected to a live tool yet.

// Prompt used
SYSTEM 
You are an operations triage assistant.
For the message below, output:
1) Category — IT/Facilities, HR/Policy, Finance/Payroll, Vendor/Procurement, or Other
2) Urgency 1-5 — anchor to business impact, not tone.
   5 = blocks payroll, payment, or a live commitment today. 1 = no deadline.
3) Sentiment — one or two words
4) A 2-3 sentence draft reply: acknowledge the issue, give a concrete next step.

Message: {input}

// Design notes
  • Urgency is anchored to business impact, not tone — note the calmly-worded printer complaint scores low while the calmly-worded payroll outage still scores 5.
  • The draft reply always names a concrete next step (a person, a timeframe) rather than a generic "we'll look into it," so a reviewer can send it with minimal editing.
  • Built to sit behind whatever intake a team already has — Microsoft Forms or Google Forms into Power Automate or n8n — rather than requiring a new tool.
PROTO-02 WORKING PROTOTYPE

AI Output Quality & Hallucination Auditor

Checks an AI-generated answer against its source text claim by claim, before it reaches a customer or stakeholder — and tells the difference between a loose approximation and a genuine hallucination.

CORE: Claude API, citation-forced scoring DESIGNED FOR: Pre-send check on any customer-facing AI reply
// Problem

Once a team uses AI to draft customer replies or policy answers, someone has to decide whether each answer is actually true to the source before it goes out — and "it reads fine" isn't a check, it's a guess.

// Designed integration
AI OUTPUT + SOURCE DOC → CLAUDE (CLAIM-BY-CLAIM AUDIT) → LOG → HUMAN REVIEW IF FLAGGED

The audit step is what's actually running below. The sheet log and reviewer routing are the designed integration.

// Prompt used
SYSTEM 
You are auditing an AI-generated answer against a source document.
Break the answer into individual factual claims.
For each claim, quote the exact source line that supports it.
If no source line supports a claim, mark it unsupported — regardless
of how confident or plausible it sounds. Distinguish a loose
approximation (close to the source, imprecise wording) from a
genuine hallucination (no basis in the source at all).

Source: {source}
AI-generated answer: {answer}
// Example run — real output for the input shown
Source document "Refund Policy (internal): Approved refunds are processed within 5 business days. Any refund above $200 requires manager sign-off before processing. Refunds are issued to the original payment method only."
AI-generated answer being audited "Refunds usually take about a week to process, and customers can choose to receive store credit instead if they prefer."
Claim 1 — "about a week to process"
Approximation
Source says 5 business days, which is roughly a week — imprecise but not unsupported. Low risk; worth tightening for accuracy, not blocking.
Claim 2 — "can choose to receive store credit instead"
Hallucination
Source explicitly states refunds go to the original payment method only. No mention of store credit anywhere — this is a confident, fabricated claim. High risk: would mislead a customer about a real option that doesn't exist.
// Design notes
  • Forces a citation before scoring — the model has to point at the exact source line behind every claim, which is what catches claim 2 above instead of letting confident phrasing pass as trustworthy.
  • Separates approximation from hallucination on purpose — they're not the same risk, and treating them the same wastes a reviewer's attention on the wrong issue.
  • Built as a pre-send check, not a post-incident one — designed to sit in front of any customer-facing AI reply, not to audit complaints after the fact.
PROTO-03 WORKING PROTOTYPE

Travel Document Extraction Bot

Pulls structured fields out of messy, inconsistently-formatted booking confirmations — built after doing this by hand for a 16-traveler conference trip and deciding it was the first thing worth automating.

CORE: Claude API, structured extraction DESIGNED FOR: Feeding a multi-traveler itinerary tracker automatically
// Problem

Flight and hotel confirmations never arrive in one format — different airlines, different booking tools, different word order — so building a group itinerary tracker means re-reading every confirmation by hand and typing the same eight fields out one at a time, every time.

// Designed integration
BOOKING TEXT → N8N → CLAUDE (STRUCTURED EXTRACTION) → TRACKER ROW → SLACK IF A FIELD IS MISSING

The extraction step is what's actually running below. The trigger, tracker, and Slack legs are the designed integration.

// Prompt used
SYSTEM 
Extract these fields from the booking text below, regardless of
its format or field order: Passenger, Booking Reference, Flight
Number, Route (origin → destination), Departure (date, time),
Arrival (date, time), Seat, Class/Baggage notes.
If a field is genuinely not present, output "Not provided" —
never guess or infer a value that isn't in the text.

Booking text: {input}
